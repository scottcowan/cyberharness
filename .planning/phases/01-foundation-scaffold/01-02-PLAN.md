<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → process | `config.toml` bytes flow into `Config`. Malformed / mistyped values cross here. |
| environment → process | Env vars override TOML. A shell that exports the wrong var name silently downgrades to defaults if `extra="forbid"` isn't set. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Tampering | config.toml keys | mitigate | `extra="forbid"` on every `BaseSettings` subclass — typos raise `ValidationError` at startup with the offending key name (RESEARCH.md Pitfall 1). |
| T-01-05 | Tampering | Invalid `mode` value | mitigate | `Literal["local","online"]` on `RoutingConfig.mode` — any other value rejected. Enforced per D-05 and CONF-04. |
| T-01-06 | Denial of Service | `probe.interval_seconds = 0` | mitigate | `Field(ge=1)` — zero or negative rejected at startup. |
| T-01-07 | Information Disclosure | Secrets in config.toml | mitigate | Schema has zero secret-shaped fields (assertion in acceptance_criteria). Future secrets are env-only per D-08. |
| T-01-08 | Information Disclosure | Env var name typo silently ignored | mitigate | `env_nested_delimiter="__"` documented in RESEARCH.md Pitfall 2. `extra="forbid"` on `Config` (root) rejects unknown envs at the root level. Nested typos still degrade silently but only affect nested keys — acceptable given closed schema. |
</threat_model>
