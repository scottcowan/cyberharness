## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → filesystem | User writes `config.toml` and env vars; harness reads. Untrusted content (typos, unexpected keys) crosses here. |
| package installer → local venv | `uv sync` installs third-party deps. Supply-chain risk crosses here. |
| user → git working tree | `.gitignore` decides what leaves the machine. Secret leakage risk. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | pyproject.toml dependency list | mitigate | All 11 runtime deps pinned with upper bounds (see Task 2 action); every package is on the approved list in RESEARCH.md Package Legitimacy Audit. Run `uv sync` in a clean env and inspect `uv.lock` before commit. |
| T-01-02 | Information Disclosure | uv.lock / .env / config.toml in git | mitigate | `.gitignore` in Task 1 covers `.env` and `.envrc`; `config.toml` lives in `~/.cyberharness/` (outside repo) so it cannot be committed accidentally; secrets rule (D-08) enforced later by config schema (no secret fields exist). |
| T-01-03 | Information Disclosure | Workspace file permissions | mitigate | Umask enforcement is Plan 03 (paths.py + startup hook). This plan does NOT create user-data files; the risk is deferred to Plan 03 by design. |
| T-01-SC | Tampering | uv/pip installs (supply-chain) | mitigate | RESEARCH.md `## Package Legitimacy Audit` covers every dep (all `[ASSUMED]` because slopcheck was unavailable at research time).
