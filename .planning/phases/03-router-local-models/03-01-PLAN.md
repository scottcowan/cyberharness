<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| harness → local OpenAI-compat endpoint | Even against localhost, `_model` and other harness-internal fields must not cross this boundary — some backends 400 on unknown top-level message keys. |
| v1.2 tool registrant → registry | Any code that registers a tool is trusted, but the *shape* of the tool spec is untrusted (may be model-suggested or copy-pasted from docs and contain the buggy `name` parameter). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | wire.py — outbound payload | mitigate | to_wire() allowlist filter — only keys in `WIRE_ALLOWED` survive; `_model` never emitted because we use attribute names, not aliases (D-09) |
| T-03-02 | Tampering / Denial of Service | tools/registry.py — tool registration | mitigate | Registration-time rejection of `name` parameter with `ToolRegistrationError` (D-08); silently-dropped tool calls surfaced as immediate errors, not runtime silent-failure |
| T-03-03 | Information Disclosure | Log leaks of message content | mitigate | to_wire() has zero logging; ToolRegistry error messages carry the tool name only, not any user content |
| T-03-SC | Tampering | `respx` dev-dep install | accept | respx is well-established httpx-native mock library (respx on PyPI, maintained; already in Phase 1 STACK.md's testing recommendations). Dev-only, no runtime footprint. See Package Legitimacy Audit in RESEARCH.md (no new runtime packages). |
</threat_model>
